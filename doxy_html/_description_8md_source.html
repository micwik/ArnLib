10 Arn Data Objects {#gen_arnobj}
11 ================
12 All objects are stored in a tree hierarchy and the naming is similar to typical
13 file systems, e.g. "//Measure/Water/Temperature/value".
14 
15 To get a handle to a folder, use a path ending with "/", e.g "//Measure/Water/".
16 
17 Folder names can be empty. In the above example, the first level folder is empty and the
18 second level folder is "Measure".
19 The empty folder name can also be referred as "@". Again, the example can equally be written
20 "/@/Measure/Water/Temperature/value". This "@" is typically used when an empty name is
21 unacceptable, e.g. in the tree viewer of the ArnBrowser tool.
22 
23 A relative path is also called the [local path](#gen_localPath), e.g.
24 "Sys/Discover/This/Service/value".
25 
26 Each part in a given path is dynamically added as needed, i.e. any path can be used without
27 explicitly creating each folder in advance.

30 ArnItem access {#gen_arnItem}
31 --------------
32 To access an _ARN Data Object_ one can use ArnM::setValue() and ArnM::valueInt() etc. This is
33 a polled access, and gives no signals / events for changed objects. Also this method is rather
34 slow as it has to locate the object via a path lookup. However its good for application assign
35 object "once".
36 
37 For continous access to an _ARN Data Object_ its better to use an ArnItem. This will be a handle
38 to the object that give fast access. It will also provide signals for changed object. ArnItem
39 is QObject based and has its charateristics.
40 
41 Yet another way to access an _ARN Data Object_, is an ArnBasicItem. This will give a basic
42 handle to the object. It is fast, small and is not based on QObject. As such it can not use
43 signals and slots, but it can provide ArnEvents.
44 
45 Normally ArnItem should be used, as it has a higher level interface with QObject signals
46 and slots. Typically ArnBasicItem is used when no signal is needed, i.e only using direct
47 access with setValue and toXXX methods.
48 If you need a lot of ArnBasicItems and memory foot print (or speed) is important, You can
49 consider to use ArnBasicItem with ArnEvents even if it will be harder to code.
50 
51 You can expect ArnBasicItem to be lees than a third of the size of an ArnItem. Tests has
52 shown ArnBasicItem to take half the time assigning an integer, compared to ArnItem.

55 Modes {#gen_arnobjModes}
56 -----
57 _Mode_ change is a one direction process. Once a specific _mode_ is set, it can't be reset.
58 
59 If the ArnItem is in a closed state when the _mode_ change is done, the added modes will
60 be stored and the real _mode_ change is done when the ArnItem is opened to an
61 _ARN Data Object_.
62 
63 If the _general mode_ change is done to a [shared](#gen_shareArnobj) object, the change
64 of _general mode_ is also done at the server and any connected clients.
65 
66 The following _general modes_ are available:
67 
68 * **BiDir** A two-way object, typically for validation or pipe.
69  See [bidirectional](#gen_bidirArnobj) objects.
70 * **Pipe** Implies _BiDir_ and all data is preserved as a stream during
71  [sharing](#gen_shareArnobj). Without _Pipe mode_, [sharing](#gen_shareArnobj) is
72  optimized to sync latest value and not all values in a stream.
73 * **Save** Sets the _ARN Data Object_ as persistent and any data assigned to it will be
74  saved. The persistent service must be started at the server.
75  See [persistent](#gen_persistArnobj) objects.
76 
77 Additionally there are some _sync modes_. These modes are used by the local client session
78 and are not shared with others.
79 The _sync modes_ must be set before the ArnItem is opened to an _ARN Data Object_.
80 
81 Following _sync_modes_ are available:
82 
83 * **Master** The _ARN Data Object_ (at client side) is set as _default generator_ of data.
84  Normally the server is the _default generator_ of data.
85  See [Sync Rules](#gen_syncRules).
86 * **AutoDestroy** The _ARN Data Object_ (at client side) is set up for auto destruction.
87  When the client closes tcp/ip, the server side will destroy the _ARN Data Object_ and
88  this will also be done at any connected clients.
89 
90 Note: It's convenient to always set all the needed modes before an ArnItem is opened or
91 an ArnItem is used as a template. See ArnItem::setTemplate().

94 Local {#gen_localPath}
95 -----
96 A relative path is also called the _local path_, e.g. the
97 [Discover remote](#gen_discoverRemote) _service name_ at path
98 "Sys/Discover/This/Service/value". The _local path_ is mapped to the absolute path
99 "/Local/". The example is then equal to "/Local/Sys/Discover/This/Service/value". The
100 _local path_ should not be [shared](#gen_shareArnobj) as it will contain specific data
101 for its running program.
102 
103 The exception to not sharing _local path_ is for some kind of remote client that must
104 be able to change an _ARN Data Object_ in the _local path_ at the remoted target.
105 For example this is used to change the [Discover remote](#gen_discoverRemote) _service name_
106 for a target host.
107 
108 Note: Do always mount the _local path_ of the server at a different path at the client.
109 This is to avoid collision with the client's own _local path_ data.
110 
111 In the above example, a remote client using ArnClient::addMountPoint("/@HostLocal/", "/Local/")
112 will share and access the [Discover remote](#gen_discoverRemote) _service name_ at the path
113 "/@HostLocal/Sys/Discover/This/Service/value".

116 Naming conventions {#gen_naming}
117 ------------------
118 These rules must not be obeyed, but are recommended, to get the most benefits of the
119 %Arn echo system, like the ArnBrowser tool.
120 
121 * First level folder empty, e.g. "//MyGlobalFolder/Date/value", is a global path and is
122  [shared](#gen_shareArnobj) to ARN server and clients.
123 * First level folder starts with "@", e.g. "/@SomeServer/MyFolder/Date/value", is a shared
124  path and is [shared](#gen_shareArnobj) to an ARN server (typically with some other
125  remote path).
126 * First level folder is "/Local", e.g "/Local/Key/value", is a [local path](#gen_localPath)
127  and is not [shared](#gen_shareArnobj).
128 * Path is relative, e.g "Key/value", is a [local path](#gen_localPath)
129  and is not [shared](#gen_shareArnobj).
130 * When a leaf is used as an attribute, the following names are reserved:
131  + **value** the value of the above closest folder denotation, e.g. "Temperature/value" (=10).
132  + **name** the describtion of the above closest folder denotation, e.g. "Server-1/name" (="Hugin").
133  + **set** allowed values and conversion to a more descriptive form, e.g. "0=Off 1=On".
134  + **bitSet** used bits and conversion to a more descriptive form, e.g. "B0=Read B1=Write".
135  + **property** like precision and unit, e.g. "prec=1 unit=°C".
136  + **info** like tool tips, e.g. "<tt>Standard UV radiation index</tt>".
137  + **help.**XXX like "help.xhtml" contains help in xhtml format.

141 Bidirectional Arn Data Objects {#gen_bidirArnobj}
142 ------------------------------
143 A bidirectional _ARN Data Object_ is actually a double object, a twin.
144 Each part has its own path but their life span is depending on each other.
145 
146 One part is the normal "official" and the other part is _provider_.
147 The provider has an added "!" to the normal path, e.g. normal = "//Measure/Depth/value",
148 provider = "//Measure/Depth/value!".
149 
150 Data written to one part ends up in the other. This can be compared to crossing electrical
151 lines from one unit to another unit regarding transmit and receive signals in each unit.
152 When a provider slot is connected to the provider part (ArnItem), the slot will receive
153 "request" data from the normal part. The provider slot processes the request data and writes
154 the result to the same provider part. This way the result will end up in the normal
155 "official" part.
156 
157 This functionality can typically be used for data validation and limiting.
158 
159 The crossing property of BiDir can be supressed by using ArnItem::setUncrossed().
160 Again this can be compared to uncrossed electrical lines from a unit to a "communicator"
161 (modem, switch, hub ...) regarding transmit and receive signals.
162 Not supprisingly this is usually a easier mode when making some kind of Bridge for ARN.

168 Pipe Arn Data Objects {#gen_pipeArnobj}
169 ---------------------
170 _Pipes_ also use the [bidirectional](#gen_bidirArnobj) functionality. The two (twin) parts
171 are then named _requester_ and _provider_.
172 
173 All data put into a pipe are part of a stream and as such will be fully transfered
174 (synchronized) if they are [shared](#gen_shareArnobj) with a server and other clients.
175 
176 ArnPipe is a specialized class for handling pipes. <Br>
177 It contains logic for handling [sequence check](#gen_pipeSeqCheck) and
178 [anti congest](#gen_pipeAntiCongest).
179 
180 Data stream to and from a pipe can be controlled using ArnItemValve class.
181 Actually ArnItemValve can controll any ArnItemB derived class.
182 <Br><Br>
183 
184 ### Pipe sequence check ### {#gen_pipeSeqCheck}
185 Sequence check is used to make sure everything is received and nothing is lost or comes twice.
186 This might happen when a tcp/ip connection goes up and down.
187 
188 The sequence check uses a hidden sequence number not visible in the pipe stream.
189 The sequence number is increased for each assignment to the pipe. The sending and checking of
190 this sequence number is activated at each end of the pipe.
191 
192 When checking is activated and the received sequence number is unexpected, a signal will be
193 generated.
194 
195 See also ArnPipe::setSendSeq(), ArnPipe::setCheckSeq(), ArnPipe::outOfSequence().
196 <Br><Br>
197 
198 ### Pipe anti congest ### {#gen_pipeAntiCongest}
199 When the pipe is a [shared oject](#gen_shareArnobj), all assignment to the pipe is queued up
200 in a send queue. If there is a disconnect in the tcp/ip, an ArnServer will drop the send queue.
201 But in an ArnClient, this send queue will grow out of control if assignments to the pipe keeps
202 coming. This problem can also arise with a fast rate of status messages on a slow network.
203 
204 One possibility is to keep track of the connection status, but this involves knowing about
205 which ArnClient (if many) to get status from. It also doesn't handle the problem with a slow
206 network.
207 
208 A probably better way is to use the _Pipe anti congest_ logic. <Br>
209 We identify _messages_ that can be sent any number of times and are used to check the data flow,
210 resending, status and alike. Typically this can be _Heart beat_, _ping_, _request update_,
211 _current time_ etc.
212 These _async messages_ are assigned using ArnPipe::setValueOverwrite().
213 
214 A regular expression is needed to identify "equal" _async messages_, that can be overwritten
215 in the send queue. If _async messages_ are repeatedly assigned to a pipe by
216 ArnPipe::setValueOverwrite(), the send queue will then not grow.
217 
218 All other _messages_ will be normally assigned to the pipe. But these _messages_ will only be
219 assigned when normal data flow is present. Typically there is some expected _feedback message_
220 from the receiving part to block uncontrolled assignment from one side of the pipe.

224 Persistent Arn Data Objects {#gen_persistArnobj}
225 ---------------------------
226 The _server_ must use ArnPersist to support the persistance service. As a standard
227 _persist storage_, _ARN Data Objects_ are stored in a SQLite database. It's also possible to
228 store each object as a file.
229 
230 The _mount point_ (path) for collecting the persistent _ARN Data objects_ is set by
231 ArnPersist::setMountPoint(). For server applications this is typically set to "/", which
232 makes all _ARN Data Objects_ potential persistent. In client applications the _mount point_
233 is typically restricted to Arn::pathLocal, which only saves local _ARN Data Objects_ in the
234 local _persist storage_.
235 
236 Any connected _client_ or the _server_ can make an _ARN Data Object_ persistent.
237 Just open an ArnItem to the object and change _mode_ to _Save_.
238 ~~~{.cpp}
239 ArnItem arnMaxLevel;
240 arnMaxLevel.addMode( Arn::ObjectMode::Save);
241 arnMaxLevel.open("//Config/Level/Max/value");
242 ~~~
243 
244 When the _ARN Data Object_ is set to _Save_ mode, it's automatically loaded by the
245 ArnPersist. At the _server_ this is instantly done. A _client_ has to wait for the value
246 to get synced from the _server_. It's convenient to use ArnDepend to get a signal
247 when the value is loaded and ready to use.
248 
249 When the _ARN Data Object_ is changed, it will be automatically saved by ArnPersist.
250 There is a delay from first change of the object until the saving is done,
251 see ArnItem::setDelay(). This allows for intensive updates of the object without choking
252 down the server with saving operations.
253 
254 It's possible to mark an object in the SQLite data base as _mandatory_. In this way the
255 _ARN Data Object_ is set as _persistent_ and gets loaded at start of ArnPersist.
256 <Br><Br>
257 
258 ### Saving objects in files ### {#gen_fileArnobj}
259 To use the _persistent_ storing of _ARN Data Objects_ in files, the _root_ directory is
260 set by: ArnPersist::setPersistDir(). This can also be combined with support of VCS
261 (version control system). See ArnPersist::setVcs(). Currently there is a support module
262 for _git_.
263 
264 In the _root_ directory and below, all (VCS) persistent files are stored.
265 The _root_ directory corresponds to the _root_ in %Arn tree.
266 
267 > Example: _root_ directory is set to "/usr/local/arn_persist". There is a file stored
268 > at "/usr/local/arn_persist/@/doc/help.xhtml". This file will be mapped to %Arn at
269 > "//doc/help.xhtml".
270 
271 Any files stored in the _root_ directory and below, get loaded into their _ARN Data Object_
272 with _mode_ set as _persistent_ at start of ArnPersist.
273 
274 The files get updated in a similar way to the data base update.

278 Sharing Arn Data Objects {#gen_shareArnobj}
279 ------------------------
280 A fundamental aspect of %Arn is that _ARN Data Objects_ can be shared. This is
281 centralized to the _ARN Server_, which stores all shared objects. It's still a
282 distributed model as each client and server has their own set of _ARN Data Objects_ that
283 operate independent of any connection.
284 
285 Each _ARN Client_ connects to the _ARN Server_ and decides which part of the
286 _ARN Data Object_ tree to be shared. <Br>
287 `ArnClient::addMountPoint("/Share/")` will make the tree "/Share/" shared. <Br>
288 This doesn't mean that everything in the shared tree at the server now will be available
289 at the client. The client has to create an _ARN Data Object_ in the shared tree.
290 The client can then decide the exact objects of interest. <Br>
291 `ArnItem::Open("/Share/Test/value")` will open a shared object in previous example.
292 
293 Note: Normally "//" or "/@.../" is used for shared. See [naming conventions](#gen_naming).
294 
295 The remote tree can be at a different path than the local tree (mount point).
296 ~~~{.cpp}
297 ArnClient::addMountPoint("/@Host/", "/") // Makes the server shared at "/@Host/".
298 ArnItem::open("/@Host/Share/Test/value") // Open the shared object in previous example.
299 ~~~

302 ### Dynamic port ### {#gen_dynamicPort}
303 An ArnServer can be created with _port_ set to 0. This will be handled as a _dynamic port_
304 and the system will assign a free _port number_ to the server. The _port number_ will be
305 taken from a range specified by IANA.
306 
307 This can typically be used to skip configuring static port numbers and be able to have
308 multiple instanses of the ArnServer on the same machine. As an ArnClient must find its
309 ArnServer, this can be used together with ArnDiscoverRemote / ArnDiscover.

313 Sync rules {#gen_syncRules}
314 ----------
315 Syncing between client and server is normally handled automatically, but for special needs
316 and reference this chapter gives an idea of the rules. Also this describes the rules when
317 connection is established. After that, normal syncing is done almost symetrically between client
318 and server.
319 
320 An _ARN Data Object_ with Master [Mode](#gen_arnobjModes) is used as _default generator_ of data.
321 Normally the server is the _default generator_ of data. This makes difference when client
322 connects or reconnects to the server. The data from the _default generator_ is then used
323 and synced.
324 
325 Also to have minimal data exchange when using non [BiDirectional](#gen_bidirArnobj)
326 _ARN Data Object_, one should take Master mode into consideration. This is more important
327 for big objects.
328 
329 When a Null value is synced, the receiver store this as an empty value, i.e. it't not stored
330 as Null which is impossible.
331 
332 ### Sync rules for Pipe ### {#gen_syncRulesPipe}
333 * Pipes should be considered to carry a flow, not a value.
334 * The pipe flow (to server) is enabled after ArnClient::connectToArn(), and is disabled
335  after ArnClient::close().
336 * In client, an enabled flow can queue up the stream of data when there is no connection
337  to server.
338 * In client, the flow keeps being enabled even if the ArnClient::connectToArn() fails
339  or there is a TCP disconnect.
340 * When the flow is disabled (ArnClient::close), all queued stream data will be sent if
341  possible.
342 * Server can never queue anything when disconnected, as the server session is only living
343  when connected.
344 <Br><Br>
345 
346 ### ClientSyncMode ### {#gen_syncRulesMode}
347 ClientSyncMode can be set with ArnClient::setSyncMode().
348 Basically this controls if a client _ARN Data Object_ is
349 considered as a Master object (se also [Modes](#gen_arnobjModes) ).
350 
351 ClientSyncMode doesn't affect a pipe. Default mode is StdAutoMaster.
352 
353 * **StdAutoMaster**
354  Dynamic auto master mode, general purpose, prohibit Null value sync.
355  Can be used for one time initial setup, thereafter server can be Master for an object.
356  + Master can be set explicitly with ArnItem::setMaster().
357  This is overided if the _ARN Data Object_ has a Null value (not assigned), then
358  the object becomes temporary Slave for next connection.
359  + If client has an unsynced local update (during not connected state), this
360  _ARN Data Object_ becomes temporary Master for just next connection.
361  + If the client is not Master for an _ARN Data Object_ but the server only has a Null
362  value, the clients value (non Null) is still used.
363 * **ImplicitMaster**
364  First local assign gives permanent Master mode, typically a client value reporter.
365  + Master can be set explicitly with ArnItem::setMaster().
366  + Client local assign to an _ARN Data Object_ gives permanent Master mode for this object.
367  This implicit Master mode setting is done once when next connection is established.
368  + Null values can be synced both from client and server.
369  + If a client _ARN Data Object_ is set booth as [Persistent](#gen_persistArnobj) and Master
370  with a Null value before connection, the Master mode is initially overridden and the
371  servers value is synced to the client.
372 * **ExplicitMaster**
373  Explicit permanent Master mode, typically an observer or manually setup Master mode.
374  Can be used for UI (User Interface) with no Master set to any _ARN Data Object_, i.e.
375  the server is always holding the "true" value.
376  + Master can be set explicitly with ArnItem::setMaster().
377  Client has no other way to become Master for an _ARN Data Object_.
378  + Null values can be synced both from client and server.
379  + If a client _ARN Data Object_ is set booth as [Persistent](#gen_persistArnobj) and Master
380  with a Null value before connection, the Master mode is initially overridden and the
381  servers value is synced to the client.

385 RPC and SAPI {#gen_rpc}
386 ============
387 ArnRpc is the basic functionality of RPC (Remote Procedure Call). ArnSapi implements SAPI
388 (Service Application Programming Interface) and is using ArnRpc as its base.
389 It's recommended to use ArnSapi which has a higher level model.
390 
391 The SAPI works by a model which can be described as RPC by _remote signal slots_.
392 The _provider_ is usually assumed to wait for a _requester_ to initiate the session
393 and then react to different remote calls from the _requester_.
394 However, this is full duplex, so any side can make a remote call at any time.
395 
396 A good example of the usage of SAPI is the "Arn Demo Chat", which is included in the
397 source package of the ArnLib.
398 
399 ArnRpc uses [pipes](\ref gen_pipeArnobj) to communicate. The _pipes_ can be monitored
400 and receive test stimuli from the "Arn Browser" program. The used
401 [protocol](#gen_rpcformat) is XString based and quite easy to handtype when common data
402 types are used. "$help" will give the syntax for the actual custom SAPI.
403 
404 A SAPI is setup by deriving the ArnSapi class to a new class that defines the
405 _custom SAPI_. This custom-declared class is included at both the _provider_ and
406 _requester_ ends.
407 The _custom SAPI_ class by itself doesn't implement any _services_. It's merely a hub for
408 connections to _external signals and slots_.
409 The base ArnSapi class automatically transfers all _custom signal_ (SAPI) calls to the
410 remote connected ends, which also have the ArnSapi derived class and that emits the
411 transfered signal. See example in ArnSapi Detailed Description.
412 
413 The provider connects the signals from custom SAPI that are prefixed with "pv_"
414 (as default) to each external slot that implements the services.
415 In the same way the _requester_ connects the signals prefixed with "rq_" to its external
416 "service" slots.
417 
418 When there is a naming pattern between the _SAPI services_ and the
419 _external signals and slots_, it's a great convenience to use ArnRpc::batchConnect(),
420 ArnSapi::batchConnectTo() or ArnSapi::batchConnectFrom().
421 This saves a lot of QObject::connect() calls.
422 Also newly added services in the SAPI, that obey the naming scheme, will automaically be
423 connected to the newly matching _external signals and slots_ for implementation of the
424 _service_.
425 
426 An extended feature comparing to normal _signals_ is that the _SAPI
427 signals_ are _public_ and can be called by non-derived classes.
428 This makes it optional to use both _signal to signal_ connections or direct _signal_ calls
429 (emit), when issuing a RPC to the remote side.
430 
431 The _service_ slot can get the emitting _custom SAPI_ object by using normal
432 QObject::sender() functionality.

435 RPC and SAPI method name overload {#gen_rpcoverload}
436 ---------------------------------
437 Under the hood Qt converts a signal that uses default argument(s) into methods with same
438 name and all variation of the arguments. I.e. One method with all arguments, one with
439 all but the last default argument, and so on until there is no more default arguments.
440 When emitting the signal with some number of arguments, all of the signal methods will
441 be exited.
442 
443 ArnRpc has to deal with this default argument mechanism, otherwise there would be multiple
444 calling messages for just one original signal emit.
445 
446 The problem arises when there also can be normal signals that are overloaded, i.e. using
447 same method name but different arguments. ArnRpc has to be able to differentiate between
448 these normal overloaded signals and the default argument signals described earlier.
449 
450 These are the alternatives, how you can help ArnRpc make your SAPI work:
451 
452 * Don't overload arguments or make sure they don't have a common start of equal names and
453  types. E.g. its ok with: `f( int a, int b); f( int b); f( int c); f( uint a);`
454 * Set ArnRpc::Mode::NoDefaultArgs and never use any default arguments in the SAPI. It's
455  then ok to use any kind of normal overloading.
456 <Br><Br>
457 
458 RPC and SAPI communication format {#gen_rpcformat}
459 ---------------------------------
460 The RPC calling has a basic format as XString (see Arn::XStringMap). A call message can
461 have 3 possible argument formats: positional, named and typed. The positional format is
462 always possible to use and is most comparable to a standard c++ call.
463 
464 The method name always come first in the message. After that comes arguments that have the
465 argument data in the value part of its key/value pair. The key part can have the argument
466 type and name, but this depends on the used argument format.
467 
468 The following RPC data types are available:
469 
470 | RPC | Qt |
471 |----------|-------------|
472 | int | int |
473 | uint | uint |
474 | int64 | qint64 |
475 | uint64 | quint64 |
476 | bool | bool |
477 | float | float |
478 | double | double |
479 | bytes | QByteArray |
480 | date | QDate |
481 | time | QTime |
482 | datetime | QDateTime |
483 | list | QStringList |
484 | string | QString |
485 
486 Also generic RPC data types can be formed as:
487 > Textual like QColor t<QColor> <Br>
488 > Binary like QPoint tb<QPoint> <Br>
489 Only textual types, i.e. those that can be converted to/from a string, are reasonable to
490 be hand typed.
491 
492 Lets have an example method to see the message when it is called.
493 > Method: void put( QString id, int value); <Br>
494 > Get called by: put("level", 123);
495 
496 Alternatives in positional argument format:
497 > put t<QString>.id=level t<int>.value=123 <Br>
498 > put string.id=level int.value=123 <Br>
499 > put string.=level int.=123 <Br>
500 > put string=level int=123 <Br>
501 > put level int=123 <Br>
502 * Argument names are optional and only for human debuging.
503 * When no type is given, "string" is asumed.
504 * When ArnRpc::Mode::NamedArg is active, its not allowed to only use typename, e.g.
505  "int=123" can be "int.=123" to enforce positional format.
506 * Both textual and binary arguments can be used.
507 
508 Alternatives in named argument format:
509 > put id=level value=123 <Br>
510 > put value=123 id=level <Br>
511 > put value=123 dummy=ABC id=level garbage=321 <Br>
512 * Only Argument names are used.
513 * Any order of arguments can be used.
514 * Extra arguments are discarded.
515 * If too few arguments, default constructor is used, e.g. "put value=123" will give id="".
516 * The methods parameter data type is used and only textual types are allowed.
517 * When ArnRpc::Mode::NamedArg is inactive, its not allowed to use an argument name that
518  also is a RPC data type. See table above. E.g. "list" and "string" are not allowed.
519 * Only textual arguments can be used (as stated before).
520 
521 Alternatives in typed argument format:
522 > put id:t<QString>=level value:t<int>=123 <Br>
523 > put id:string=level value:int=123 <Br>
524 > put value:int=123 id:string=level <Br>
525 > put value:int=123 dummy:bytes=ABC id:string=level <Br>
526 * Argument names and types are used.
527 * Only the name is used to match method parameter.
528 * The type is verified with the matching method parameter for error check.
529 * Any order of arguments can be used.
530 * Extra arguments are discarded.
531 * If too few arguments, default constructor is used, e.g. "put value:int=123" will give id="".
532 * Both textual and binary arguments can be used.
533 
534 Named and typed argument format can be mixed, but positional format is never mixed.
535 
536 List (QStringList) can be used. All examples below will get
537 same resulting call.
538 > For a function: void test( QStringList lst, int num) <Br>
539 > test list=red green blu int=3 <Br>
540 > test list.lst=red green blu int.num=3 <Br>
541 > test list= +=red +==green +=blu int=3 <Br>
542 > test list=red +=green blu int=3 <Br>
543 > test lst:list=red green blu num=3 <Br>
544 > test num=3 lst:list=red green +=blu <Br>
545 * list is both a data type and a syntax for defining its data.
546 * list is only available for positional and typed argument format.
547 
548 For special cases, like empty elements, the += syntax is needed. The example
549 below has a first empty element followed by "green".
550 > test list= += green blue int=2
551 
552 The built-in call "$help" will give an automatically generated list of the present SAPI
553 with the syntax for each available service. The default argument format is positional.
554 This can be changed to named format by giving "$help named".

558 ZeroConfig {#gen_zeroconf}
559 ==========
560 For getting a basic understanding of ZeroConfig and further references to relevant
561 documentation, see: http://zeroconf.org/
562 
563 _ARN ZeroConfig_ is the lowest level support for advertising and discovering services on
564 a local network. The implementation has very few dependencies to the rest of the ArnLib.
565 
566 _ARN ZeroConfig_ can use a built in implementation of Apple (R) _mDns_ / _DNS_SD_ that has no
567 further dependencies to external libraries. For _mDns_ the low end system abstraction layer
568 has been written to use Qt for portability. The higher level _DNS_SD_ has wrappers written
569 to give a good c++ / Qt API.
570 
571 It's also possible to use an external _DNS_SD_ library, like _Avahi_. This gives better
572 performance when many applications uses ZeroConfig on the same machine, as they share
573 cashing etc with a common daemon. However you have to deal with this external dependency.
574 
575 _ARN ZeroConfig_ implementation has two parts. The ArnZeroConfRegister can be used to
576 advertise any _service_ given a _host address_ and a _port number_. The other part is the
577 ArnZeroConfBrowser / ArnZeroConfResolve / ArnZeroConfLookup. The browser is used to get a
578 realtime list of available _services_ on the network. The resolver takes a given _service_
579 and resolves it into its _host name_ and _port number_. Finally ArnZeroConfLookup takes
580 a given _host name_ and makes a DNS (mDNS) lookup to get its ip-address. Each of these
581 classes are stand alone and has to be combined with glue logic for the complete process.
582 
583 ZeroConfig definitions {#gen_zeroconfDef}
584 ----------------------
585 A ZeroConfig _service_ has a _service type_, that preferably should be registered at IANA.
586 Examples of _service types_ are "http", "ftp" and "arn". This type is mandatory when
587 advertising a _service_. Also the _service_ must have a _service name_.
588 <Br><Br>
589 
590 ### Service name ### {#gen_zeroconfServiceName}
591 
592 _Service names_ can be any human readable id. It should be easy to understand, without
593 any cryptic coding. There should not be any attempts to make the _service name_ unique
594 as this is taken care of by the ZeroConfig system. It's common that the _service name_
595 can be modified by the end user. The default starting name could be some system or product
596 name. Example of _service name_: "My House Registry".
597 <Br><Br>
598 
599 ### Sub types ### {#gen_zeroconfSubTypes}
600 _Services_ can also have _sub types_. These are identifiers that can be used to filter out
601 some sub group from a specific _service type_. All _services_ having the same _service type_
602 must still have some common protocol even if they belong to different _sub types_.
603 A _service_ can be advertised with many _sub types_, but browsing can only be filtered
604 with one _sub type_ or with no filter.
605 <Br><Br>
606 
607 ### Text record ### {#gen_zeroconfTextRecord}
608 It's possible to add a _text record_ to a _service_. The format of this record is specified
609 by IANA. The purpose is to store properties by a _key_ / _value_ -pair. For convenience
610 this can be done with ArnZeroConfRegister::setTxtRecordMap() using an Arn::XStringMap.
611 <Br><Br>

614 Discover {#gen_discover}
615 --------
616 _ARN Discover_ is the mid level support for advertising and discovering services on
617 a local network. This implementation is only for the "arn" _service type_ and is heavily
618 dependent on the ArnLib. The "arn" _service type_ is approved and registered by IANA.
619 
620 _ARN Discover_ implementation has two parts. The ArnDiscoverAdvertise can be used to
621 advertise an %Arn _service_ given a _host address_ and a _port number_. The other part is
622 the ArnDiscoverBrowser / ArnDiscoverResolver. The browser is used to get a realtime list
623 of available %Arn _services_ on the network. The resolver is for taking a manual resolve
624 when a _service name_ is known in advance.
625 
626 _ARN Discover_ is designed to minimize external glue logic as these classes do all the
627 common processing. Internally _ARN ZeroConfig_ is used, but focus is on solving %Arn
628 specific needs in a powerful, yet flexible manner.
629 
630 An _ARN service_ needs an ArnDiscover::Type and a [service name](\ref gen_zeroconfServiceName).
631 The ArnDiscover::Type sets up a coarse division of the applications into the _groups_
632 "server" and "client". The "client" typically only offer the service of ArnDiscoverRemote.
633 
634 _ARN services_ can also have _groups_. These are identifiers that can be used to filter out
635 some sub group. An _ARN service_ can be advertised with many _groups_, but browsing can only be
636 filtered with one _group_ or with no filter.
637 
638 It's possible to add a _custom property_ to an _ARN service_. This can be done with
639 ArnDiscoverAdvertise::setCustomProperties() using an Arn::XStringMap. The propertie has a
640 _key_ / _value_ -pair. The custom property are advised to have a _key_ starting with a
641 capital letter to avoid name collision with the system.
642 The added _groups_ will be set as properties with naming as "group0", "group1" ...
643 
644 ArnDiscoverBrowser collects found %Arn _services_. Each of these _services_ can automatically
645 be further examined. This is chosen by calling ArnDiscoverBrowserB::setDefaultStopState(),
646 which e.g. tells examination to stop after _host name_ has been found. The _service_ can
647 then manually be ordered for further examination by ArnDiscoverBrowserB::goTowardState(),
648 e.g. examination should now stop after _host ip_ is found.
649 
650 All the information about a _service_ is stored in ArnDiscoverInfo. Found _services_ can
651 be accessed by index, id or _service name_. Increasing index, starting at 0, gives a list
652 of _services_ alfabetically sorted by _service name_. The index is kind of volatile and
653 should be used instantly, not be stored. The id gives a unique number for each service and
654 can be stored. However the _service_ given by the id might dissapear.
655 <Br><Br>

658 Discover remote {#gen_discoverRemote}
659 ---------------
660 _ARN Discover Remote_ is the highest level support for advertising and discovering services
661 on a local network. Its implementation is based on _ARN Discover_. The added functionality
662 is to have a remote control for both advertising an ArnServer and multiple ArnClient
663 connections. The remote control is done via _ARN Data Objects_ in [local path](#gen_localPath)
664 "Sys/Discover/".
665 
666 _ARN Discover Remote_ has one main class, ArnDiscoverRemote which act as a central point.
667 The ArnDiscoverRemote class also takes an ArnServer and advertises it as a _service_. For
668 remote control the _service name_ is available at [local path](#gen_localPath)
669 "Sys/Discover/This/Service/value".
670 
671 ArnDiscoverRemote can make an internal ArnServer, when there is no need to access the
672 ArnServer class. This is usually the case in an client application. The ArnServer is then
673 merely used to make the discover functionality remote controlled.
674 
675 Remote controlled client connections can be added. Each ArnClient is handled by an
676 ArnDiscoverConnector instance, which is made by ArnDiscoverRemote::newConnector().
677 Connections can be added to ArnDiscoverConnector, both as a _direct host_ list and a
678 _discover host_.
679 
680 The _discover host_ is indirerctly set, by adding an ArnDiscoverResolver to
681 ArnDiscoverConnector. A _service name_ can then be resolved into the _discover host_.
682 
683 The two connection methods can coexist and as standard the _discover host_ has lower
684 priority number than _direct host_, i.e. _discover host_ is tried first.
685 
686 The ArnDiscoverConnector is associated with an _id_, which should be chosen to describe the
687 client target or its purpose. It's not a host address or necessarily a specific host, as
688 there can be many possible addresses assigned to the ArnDiscoverConnector.
689 
690 The _id_ will appear as an _ARN folder_ in [local path](#gen_localPath), e.g. when _id_ is
691 "WeatherData-XYZ" the folder path will be "Sys/Discover/Connect/WeatherData-XYZ/". The
692 folder and its sub folders will contain _ARN Data Objects_ to remote control the ArnClient.
693 For a more comprehensive description of these objects, see
694 [help discover description](@ref helpDiscDiscover).
695 
696 In the above example, a _discover host_ can be remote controlled by setting the
697 _service name_ in [local path](#gen_localPath)
698 "Sys/Discover/Connect/WeatherData-XYZ/DiscoverHost/Service/value", e.g. to
699 "Region Weather XYZ".
700 
701 Also in the above example, the first _direct host_ can be remote controlled by setting the
702 _host name_ in [local path](#gen_localPath)
703 "Sys/Discover/Connect/WeatherData-XYZ/DirectHosts/Host-0/value", e.g. to "localhost".
704 
705 Normally it's wanted that any remote set values in the [local path](#gen_localPath) remains
706 after power cycling. This is supported by the [Arn persist system](#gen_persistArnobj).
707 
708 Connecting via resolver uses the logic:
709 * If connection fails for a _discover host_, resolving is forced to be refreshed for the
710  target _service name_. The Host for the _service name_ might have changed since last
711  resolved and doing a refresh can get the new _discover host_.
712 
713 * If connection continues to fail for a _discover host_, refreshing the resolv will have a
714  blocking time to avoid spamming the net. Typically this time is 30 seconds, but it can be
715  changed by ArnDiscoverConnector::setResolveRefreshTimeout().
