10 Arn Data Objects {#gen_arnobj}
11 ================
12 All objects are stored in a tree hierarchy and the naming is similar to typical
13 file systems, e.g. "//Measure/Water/Temperature/value".
14 
15 To get a handle to a folder, use a path ending with "/", e.g "//Measure/Water/".
16 
17 Folder names can be empty. In the above example, the first level folder is empty and the
18 second level folder is "Measure".
19 The empty folder name can also be referred as "@". Again, the example can equally be written
20 "/@/Measure/Water/Temperature/value". This "@" is typically used when an empty name is
21 unacceptable, e.g. in the tree viewer of the ArnBrowser tool.
22 
23 A relative path is also called the [local path](#gen_localPath), e.g.
24 "Sys/Discover/This/Service/value".
25 
26 Each part in a given path is dynamically added as needed, i.e. any path can be used without
27 explicitly creating each folder in advance.

30 ArnItem access {#gen_arnItem}
31 --------------
32 To access an _ARN Data Object_ one can use ArnM::setValue() and ArnM::valueInt() etc. This is
33 a polled access, and gives no signals / events for changed objects. Also this method is rather
34 slow as it has to locate the object via a path lookup. However its good for application assign
35 object "once".
36 
37 For continous access to an _ARN Data Object_ its better to use an ArnItem. This will be a handle
38 to the object that give fast access. It will also provide signals for changed object. ArnItem
39 is QObject based and has its charateristics.
40 
41 Yet another way to access an _ARN Data Object_, is an ArnBasicItem. This will give a a basic
42 handle to the object. It is fast, small and is not based on QObject. As such it can not use
43 signals and slots, but it can provide ArnEvents.
44 
45 Normally ArnItem should be used, as it has a higher level interface with QObject signals
46 and slots. Typically ArnBasicItem is used when no signal is needed, i.e only using direct
47 access with setValue and toXXX methods.
48 If you need a lot of ArnBasicItems and memory foot print (or speed) is important, You can
49 consider to use ArnBasicItem with ArnEvents even if it will be harder to code.
50 
51 You can expect ArnBasicItem to be lees than a third of the size of an ArnItem. Tests has
52 shown ArnBasicItem to take half the time assigning an integer, compared to ArnItem.

55 Modes {#gen_arnobjModes}
56 -----
57 _Mode_ change is a one direction process. Once a specific _mode_ is set, it can't be reset.
58 
59 If the ArnItem is in a closed state when the _mode_ change is done, the added modes will
60 be stored and the real _mode_ change is done when the ArnItem is opened to an
61 _ARN Data Object_.
62 
63 If the _general mode_ change is done to a [shared](#gen_shareArnobj) object, the change
64 of _general mode_ is also done at the server and any connected clients.
65 
66 The following _general modes_ are available:
67 
68 * **BiDir** A two-way object, typically for validation or pipe.
69  See [bidirectional](#gen_bidirArnobj) objects.
70 * **Pipe** Implies _BiDir_ and all data is preserved as a stream during
71  [sharing](#gen_shareArnobj). Without _Pipe mode_, [sharing](#gen_shareArnobj) is
72  optimized to sync latest value and not all values in a stream.
73 * **Save** Sets the _ARN Data Object_ as persistent and any data assigned to it will be
74  saved. The persistent service must be started at the server.
75  See [persistent](#gen_persistArnobj) objects.
76 
77 Additionally there are some _sync modes_. These modes are used by the local client session
78 and are not shared with others.
79 The _sync modes_ must be set before the ArnItem is opened to an _ARN Data Object_.
80 
81 Following _sync_modes_ are available:
82 
83 * **Master** The _ARN Data Object_ (at client side) is set as _default generator_ of data.
84  Normally the server is the _default generator_ of data.
85  This makes difference when client connects or reconnects to the server.
86  The data from the _default generator_ is then used and synced. Also echo of data to
87  the client side _ARN data object_ is prohibited.
88 * **AutoDestroy** The _ARN Data Object_ (at client side) is set up for auto destruction.
89  When the client closes tcp/ip, the server side will destroy the _ARN Data Object_ and
90  this will also be done at any connected clients.
91 
92 Note: It's convenient to always set all the needed modes before an ArnItem is opened or
93 an ArnItem is used as a template. See ArnItem::setTemplate().

96 Local {#gen_localPath}
97 -----
98 A relative path is also called the _local path_, e.g. the
99 [Discover remote](#gen_discoverRemote) _service name_ at path
100 "Sys/Discover/This/Service/value". The _local path_ is mapped to the absolute path
101 "/Local/". The example is then equal to "/Local/Sys/Discover/This/Service/value". The
102 _local path_ should not be [shared](#gen_shareArnobj) as it will contain specific data
103 for its running program.
104 
105 The exception to not sharing _local path_ is for some kind of remote client that must
106 be able to change an _ARN Data Object_ in the _local path_ at the remoted target.
107 For example this is used to change the [Discover remote](#gen_discoverRemote) _service name_
108 for a target host.
109 
110 Note: Do always mount the _local path_ of the server at a different path at the client.
111 This is to avoid collision with the client's own _local path_ data.
112 
113 In the above example, a remote client using ArnClient::addMountPoint("/@HostLocal/", "/Local/")
114 will share and access the [Discover remote](#gen_discoverRemote) _service name_ at the path
115 "/@HostLocal/Sys/Discover/This/Service/value".

118 Naming conventions {#gen_naming}
119 ------------------
120 These rules must not be obeyed, but are recommended, to get the most benefits of the
121 %Arn echo system, like the ArnBrowser tool.
122 
123 * First level folder empty, e.g. "//MyGlobalFolder/Date/value", is a global path and is
124  [shared](#gen_shareArnobj) to ARN server and clients.
125 * First level folder starts with "@", e.g. "/@SomeServer/MyFolder/Date/value", is a shared
126  path and is [shared](#gen_shareArnobj) to an ARN server (typically with some other
127  remote path).
128 * First level folder is "/Local", e.g "/Local/Key/value", is a [local path](#gen_localPath)
129  and is not [shared](#gen_shareArnobj).
130 * Path is relative, e.g "Key/value", is a [local path](#gen_localPath)
131  and is not [shared](#gen_shareArnobj).
132 * When a leaf is used as an attribute, the following names are reserved:
133  + **value** the value of the above closest folder denotation, e.g. "Temperature/value" (=10).
134  + **name** the describtion of the above closest folder denotation, e.g. "Server-1/name" (="Hugin").
135  + **set** allowed values and conversion to a more descriptive form, e.g. "0=Off 1=On".
136  + **bitSet** used bits and conversion to a more descriptive form, e.g. "B0=Read B1=Write".
137  + **property** like precision and unit, e.g. "prec=1 unit=Â°C".
138  + **info** like tool tips, e.g. "<tt>Standard UV radiation index</tt>".
139  + **help.**XXX like "help.xhtml" contains help in xhtml format.

143 Bidirectional Arn Data Objects {#gen_bidirArnobj}
144 ------------------------------
145 A bidirectional _ARN Data Object_ is actually a double object, a twin.
146 Each part has its own path but their life span is depending on each other.
147 
148 One part is the normal "official" and the other part is _provider_.
149 The provider has an added "!" to the normal path, e.g. normal = "//Measure/Depth/value",
150 provider = "//Measure/Depth/value!".
151 
152 Data written to one part ends up in the other. When a provider slot is connected to the
153 provider part (ArnItem), the slot will receive "request" data from the normal part.
154 The provider slot processes the request data and writes the result to the same provider part.
155 This way the result will end up in the normal "official" part.
156 
157 This functionality can typically be used for data validation and limiting.

161 Pipe Arn Data Objects {#gen_pipeArnobj}
162 ---------------------
163 _Pipes_ also use the [bidirectional](#gen_bidirArnobj) functionality. The two (twin) parts
164 are then named _requester_ and _provider_.
165 
166 All data put into a pipe are part of a stream and as such will be fully transfered
167 (synchronized) if they are [shared](#gen_shareArnobj) with a server and other clients.
168 
169 ArnPipe is a specialized class for handling pipes. <Br>
170 It contains logic for handling [sequence check](#gen_pipeSeqCheck) and
171 [anti congest](#gen_pipeAntiCongest).
172 
173 Data stream to and from a pipe can be controlled using ArnItemValve class.
174 Actually ArnItemValve can controll any ArnItemB derived class.
175 <Br><Br>
176 
177 ### Pipe sequence check ### {#gen_pipeSeqCheck}
178 Sequence check is used to make sure everything is received and nothing is lost or comes twice.
179 This might happen when a tcp/ip connection goes up and down.
180 
181 The sequence check uses a hidden sequence number not visible in the pipe stream.
182 The sequence number is increased for each assignment to the pipe. The sending and checking of
183 this sequence number is activated at each end of the pipe.
184 
185 When checking is activated and the received sequence number is unexpected, a signal will be
186 generated.
187 
188 See also ArnPipe::setSendSeq(), ArnPipe::setCheckSeq(), ArnPipe::outOfSequence().
189 <Br><Br>
190 
191 ### Pipe anti congest ### {#gen_pipeAntiCongest}
192 When the pipe is a [shared oject](#gen_shareArnobj), all assignment to the pipe is queued up
193 in a send queue. If there is a disconnect in the tcp/ip, an ArnServer will drop the send queue.
194 But in an ArnClient, this send queue will grow out of control if assignments to the pipe keeps
195 coming. This problem can also arise with a fast rate of status messages on a slow network.
196 
197 One possibility is to keep track of the connection status, but this involves knowing about
198 which ArnClient (if many) to get status from. It also doesn't handle the problem with a slow
199 network.
200 
201 A probably better way is to use the _Pipe anti congest_ logic. <Br>
202 We identify _messages_ that can be sent any number of times and are used to check the data flow,
203 resending, status and alike. Typically this can be _Heart beat_, _ping_, _request update_,
204 _current time_ etc.
205 These _async messages_ are assigned using ArnPipe::setValueOverwrite().
206 
207 A regular expression is needed to identify "equal" _async messages_, that can be overwritten
208 in the send queue. If _async messages_ are repeatedly assigned to a pipe by
209 ArnPipe::setValueOverwrite(), the send queue will then not grow.
210 
211 All other _messages_ will be normally assigned to the pipe. But these _messages_ will only be
212 assigned when normal data flow is present. Typically there is some expected _feedback message_
213 from the receiving part to block uncontrolled assignment from one side of the pipe.

217 Persistent Arn Data Objects {#gen_persistArnobj}
218 ---------------------------
219 The _server_ must use ArnPersist to support the persistance service. As a standard
220 _persist storage_, _ARN Data Objects_ are stored in a SQLite database. It's also possible to
221 store each object as a file.
222 
223 The _mount point_ (path) for collecting the persistent _ARN Data objects_ is set by
224 ArnPersist::setMountPoint(). For server applications this is typically set to "/", which
225 makes all _ARN Data Objects_ potential persistent. In client applications the _mount point_
226 is typically restricted to Arn::pathLocal, which only saves local _ARN Data Objects_ in the
227 local _persist storage_.
228 
229 Any connected _client_ or the _server_ can make an _ARN Data Object_ persistent.
230 Just open an ArnItem to the object and change _mode_ to _Save_.
231 ~~~{.cpp}
232 ArnItem arnMaxLevel;
233 arnMaxLevel.addMode( Arn::ObjectMode::Save);
234 arnMaxLevel.open("//Config/Level/Max/value");
235 ~~~
236 
237 When the _ARN Data Object_ is set to _Save_ mode, it's automatically loaded by the
238 ArnPersist. At the _server_ this is instantly done. A _client_ has to wait for the value
239 to get synced from the _server_. It's convenient to use ArnDepend to get a signal
240 when the value is loaded and ready to use.
241 
242 When the _ARN Data Object_ is changed, it will be automatically saved by ArnPersist.
243 There is a delay from first change of the object until the saving is done,
244 see ArnItem::setDelay(). This allows for intensive updates of the object without choking
245 down the server with saving operations.
246 
247 It's possible to mark an object in the SQLite data base as _mandatory_. In this way the
248 _ARN Data Object_ is set as _persistent_ and gets loaded at start of ArnPersist.
249 <Br><Br>
250 
251 ### Saving objects in files ### {#gen_fileArnobj}
252 To use the _persistent_ storing of _ARN Data Objects_ in files, the _root_ directory is
253 set by: ArnPersist::setPersistDir(). This can also be combined with support of VCS
254 (version control system). See ArnPersist::setVcs(). Currently there is a support module
255 for _git_.
256 
257 In the _root_ directory and below, all (VCS) persistent files are stored.
258 The _root_ directory corresponds to the _root_ in %Arn tree.
259 
260 > Example: _root_ directory is set to "/usr/local/arn_persist". There is a file stored
261 > at "/usr/local/arn_persist/@/doc/help.xhtml". This file will be mapped to %Arn at
262 > "//doc/help.xhtml".
263 
264 Any files stored in the _root_ directory and below, get loaded into their _ARN Data Object_
265 with _mode_ set as _persistent_ at start of ArnPersist.
266 
267 The files get updated in a similar way to the data base update.

271 Sharing Arn Data Objects {#gen_shareArnobj}
272 ------------------------
273 A fundamental aspect of %Arn is that _ARN Data Objects_ can be shared. This is
274 centralized to the _ARN Server_, which stores all shared objects. It's still a
275 distributed model as each client and server has their own set of _ARN Data Objects_ that
276 operate independent of any connection.
277 
278 Each _ARN Client_ connects to the _ARN Server_ and decides which part of the
279 _ARN Data Object_ tree to be shared. <Br>
280 `ArnClient::addMountPoint("/Share/")` will make the tree "/Share/" shared. <Br>
281 This doesn't mean that everything in the shared tree at the server now will be available
282 at the client. The client has to create an _ARN Data Object_ in the shared tree.
283 The client can then decide the exact objects of interest. <Br>
284 `ArnItem::Open("/Share/Test/value")` will open a shared object in previous example.
285 
286 Note: Normally "//" or "/@.../" is used for shared. See [naming conventions](#gen_naming).
287 
288 The remote tree can be at a different path than the local tree (mount point).
289 ~~~{.cpp}
290 ArnClient::addMountPoint("/@Host/", "/") // Makes the server shared at "/@Host/".
291 ArnItem::open("/@Host/Share/Test/value") // Open the shared object in previous example.
292 ~~~

295 ### Dynamic port ### {#gen_dynamicPort}
296 An ArnServer can be created with _port_ set to 0. This will be handled as a _dynamic port_
297 and the system will assign a free _port number_ to the server. The _port number_ will be
298 taken from a range specified by IANA.
299 
300 This can typically be used to skip configuring static port numbers and be able to have
301 multiple instanses of the ArnServer on the same machine. As an ArnClient must find its
302 ArnServer, this can be used together with ArnDiscoverRemote / ArnDiscover.

306 RPC and SAPI {#gen_rpc}
307 ============
308 ArnRpc is the basic functionality of RPC (Remote Procedure Call). ArnSapi implements SAPI
309 (Service Application Programming Interface) and is using ArnRpc as its base.
310 It's recommended to use ArnSapi which has a higher level model.
311 
312 The SAPI works by a model which can be described as RPC by _remote signal slots_.
313 The _provider_ is usually assumed to wait for a _requester_ to initiate the session
314 and then react to different remote calls from the _requester_.
315 However, this is full duplex, so any side can make a remote call at any time.
316 
317 A good example of the usage of SAPI is the "Arn Demo Chat", which is included in the
318 source package of the ArnLib.
319 
320 ArnRpc uses [pipes](\ref gen_pipeArnobj) to communicate. The _pipes_ can be monitored
321 and receive test stimuli from the "Arn Browser" program. The used
322 [protocol](#gen_rpcformat) is XString based and quite easy to handtype when common data
323 types are used. "$help" will give the syntax for the actual custom SAPI.
324 
325 A SAPI is setup by deriving the ArnSapi class to a new class that defines the
326 _custom SAPI_. This custom-declared class is included at both the _provider_ and
327 _requester_ ends.
328 The _custom SAPI_ class by itself doesn't implement any _services_. It's merely a hub for
329 connections to _external signals and slots_.
330 The base ArnSapi class automatically transfers all _custom signal_ (SAPI) calls to the
331 remote connected ends, which also have the ArnSapi derived class and that emits the
332 transfered signal. See example in ArnSapi Detailed Description.
333 
334 The provider connects the signals from custom SAPI that are prefixed with "pv_"
335 (as default) to each external slot that implements the services.
336 In the same way the _requester_ connects the signals prefixed with "rq_" to its external
337 "service" slots.
338 
339 When there is a naming pattern between the _SAPI services_ and the
340 _external signals and slots_, it's a great convenience to use ArnRpc::batchConnect(),
341 ArnSapi::batchConnectTo() or ArnSapi::batchConnectFrom().
342 This saves a lot of QObject::connect() calls.
343 Also newly added services in the SAPI, that obey the naming scheme, will automaically be
344 connected to the newly matching _external signals and slots_ for implementation of the
345 _service_.
346 
347 An extended feature comparing to normal _signals_ is that the _SAPI
348 signals_ are _public_ and can be called by non-derived classes.
349 This makes it optional to use both _signal to signal_ connections or direct _signal_ calls
350 (emit), when issuing a RPC to the remote side.
351 
352 The _service_ slot can get the emitting _custom SAPI_ object by using normal
353 QObject::sender() functionality.

356 RPC and SAPI method name overload {#gen_rpcoverload}
357 ---------------------------------
358 Under the hood Qt converts a signal that uses default argument(s) into methods with same
359 name and all variation of the arguments. I.e. One method with all arguments, one with
360 all but the last default argument, and so on until there is no more default arguments.
361 When emitting the signal with some number of arguments, all of the signal methods will
362 be exited.
363 
364 ArnRpc has to deal with this default argument mechanism, otherwise there would be multiple
365 calling messages for just one original signal emit.
366 
367 The problem arises when there also can be normal signals that are overloaded, i.e. using
368 same method name but different arguments. ArnRpc has to be able to differentiate between
369 these normal overloaded signals and the default argument signals described earlier.
370 
371 These are the alternatives, how you can help ArnRpc make your SAPI work:
372 
373 * Don't overload arguments or make sure they don't have a common start of equal names and
374  types. E.g. its ok with: `f( int a, int b); f( int b); f( int c); f( uint a);`
375 * Set ArnRpc::Mode::NoDefaultArgs and never use any default arguments in the SAPI. It's
376  then ok to use any kind of normal overloading.
377 <Br><Br>

379 RPC and SAPI communication format {#gen_rpcformat}
380 ---------------------------------
381 The RPC calling has a basic format as XString (see Arn::XStringMap). A call message can
382 have 3 possible argument formats: positional, named and typed. The positional format is
383 always possible to use and is most comparable to a standard c++ call.
384 
385 The method name always come first in the message. After that comes arguments that have the
386 argument data in the value part of its key/value pair. The key part can have the argument
387 type and name, but this depends on the used argument format.
388 
389 The following RPC data types are available:
390 
391 | RPC | Qt |
392 |----------|-------------|
393 | int | int |
394 | uint | uint |
395 | int64 | qint64 |
396 | uint64 | quint64 |
397 | bool | bool |
398 | float | float |
399 | double | double |
400 | bytes | QByteArray |
401 | date | QDate |
402 | time | QTime |
403 | datetime | QDateTime |
404 | list | QStringList |
405 | string | QString |
406 
407 Also generic RPC data types can be formed as:
408 > Textual like QColor t<QColor> <Br>
409 > Binary like QPoint tb<QPoint> <Br>
410 Only textual types, i.e. those that can be converted to/from a string, are reasonable to
411 be hand typed.
412 
413 Lets have an example method to see the message when it is called.
414 > Method: void put( QString id, int value); <Br>
415 > Get called by: put("level", 123);
416 
417 Alternatives in positional argument format:
418 > put t<QString>.id=level t<int>.value=123 <Br>
419 > put string.id=level int.value=123 <Br>
420 > put string.=level int.=123 <Br>
421 > put string=level int=123 <Br>
422 > put level int=123 <Br>
423 * Argument names are optional and only for human debuging.
424 * When no type is given, "string" is asumed.
425 * When ArnRpc::Mode::NamedArg is active, its not allowed to only use typename, e.g.
426  "int=123" can be "int.=123" to enforce positional format.
427 * Both textual and binary arguments can be used.
428 
429 Alternatives in named argument format:
430 > put id=level value=123 <Br>
431 > put value=123 id=level <Br>
432 > put value=123 dummy=ABC id=level garbage=321 <Br>
433 * Only Argument names are used.
434 * Any order of arguments can be used.
435 * Extra arguments are discarded.
436 * If too few arguments, default constructor is used, e.g. "put value=123" will give id="".
437 * The methods parameter data type is used and only textual types are allowed.
438 * When ArnRpc::Mode::NamedArg is inactive, its not allowed to use an argument name that
439  also is a RPC data type. See table above. E.g. "list" and "string" are not allowed.
440 * Only textual arguments can be used (as stated before).
441 
442 Alternatives in typed argument format:
443 > put id:t<QString>=level value:t<int>=123 <Br>
444 > put id:string=level value:int=123 <Br>
445 > put value:int=123 id:string=level <Br>
446 > put value:int=123 dummy:bytes=ABC id:string=level <Br>
447 * Argument names and types are used.
448 * Only the name is used to match method parameter.
449 * The type is verified with the matching method parameter for error check.
450 * Any order of arguments can be used.
451 * Extra arguments are discarded.
452 * If too few arguments, default constructor is used, e.g. "put value:int=123" will give id="".
453 * Both textual and binary arguments can be used.
454 
455 Named and typed argument format can be mixed, but positional format is never mixed.
456 
457 List (QStringList) can be used. All examples below will get
458 same resulting call.
459 > For a function: void test( QStringList lst, int num) <Br>
460 > test list=red green blu int=3 <Br>
461 > test list.lst=red green blu int.num=3 <Br>
462 > test list= +=red +==green +=blu int=3 <Br>
463 > test list=red +=green blu int=3 <Br>
464 > test lst:list=red green blu num=3 <Br>
465 > test num=3 lst:list=red green +=blu <Br>
466 * list is both a data type and a syntax for defining its data.
467 * list is only available for positional and typed argument format.
468 
469 For special cases, like empty elements, the += syntax is needed. The example
470 below has a first empty element followed by "green".
471 > test list= += green blue int=2
472 
473 The built-in call "$help" will give an automatically generated list of the present SAPI
474 with the syntax for each available service. The default argument format is positional.
475 This can be changed to named format by giving "$help named".

479 ZeroConfig {#gen_zeroconf}
480 ==========
481 For getting a basic understanding of ZeroConfig and further references to relevant
482 documentation, see: http://zeroconf.org/
483 
484 _ARN ZeroConfig_ is the lowest level support for advertising and discovering services on
485 a local network. The implementation has very few dependencies to the rest of the ArnLib.
486 
487 _ARN ZeroConfig_ can use a built in implementation of Apple (R) _mDns_ / _DNS_SD_ that has no
488 further dependencies to external libraries. For _mDns_ the low end system abstraction layer
489 has been written to use Qt for portability. The higher level _DNS_SD_ has wrappers written
490 to give a good c++ / Qt API.
491 
492 It's also possible to use an external _DNS_SD_ library, like _Avahi_. This gives better
493 performance when many applications uses ZeroConfig on the same machine, as they share
494 cashing etc with a common daemon. However you have to deal with this external dependency.
495 
496 _ARN ZeroConfig_ implementation has two parts. The ArnZeroConfRegister can be used to
497 advertise any _service_ given a _host address_ and a _port number_. The other part is the
498 ArnZeroConfBrowser / ArnZeroConfResolve / ArnZeroConfLookup. The browser is used to get a
499 realtime list of available _services_ on the network. The resolver takes a given _service_
500 and resolves it into its _host name_ and _port number_. Finally ArnZeroConfLookup takes
501 a given _host name_ and makes a DNS (mDNS) lookup to get its ip-address. Each of these
502 classes are stand alone and has to be combined with glue logic for the complete process.
503 
504 ZeroConfig definitions {#gen_zeroconfDef}
505 ----------------------
506 A ZeroConfig _service_ has a _service type_, that preferably should be registered at IANA.
507 Examples of _service types_ are "http", "ftp" and "arn". This type is mandatory when
508 advertising a _service_. Also the _service_ must have a _service name_.

511 ### Service name ### {#gen_zeroconfServiceName}
512 
513 _Service names_ can be any human readable id. It should be easy to understand, without
514 any cryptic coding. There should not be any attempts to make the _service name_ unique
515 as this is taken care of by the ZeroConfig system. It's common that the _service name_
516 can be modified by the end user. The default starting name could be some system or product
517 name. Example of _service name_: "My House Registry".
518 <Br><Br>
519 
520 ### Sub types ### {#gen_zeroconfSubTypes}
521 _Services_ can also have _sub types_. These are identifiers that can be used to filter out
522 some sub group from a specific _service type_. All _services_ having the same _service type_
523 must still have some common protocol even if they belong to different _sub types_.
524 A _service_ can be advertised with many _sub types_, but browsing can only be filtered
525 with one _sub type_ or with no filter.
526 <Br><Br>
527 
528 ### Text record ### {#gen_zeroconfTextRecord}
529 It's possible to add a _text record_ to a _service_. The format of this record is specified
530 by IANA. The purpose is to store properties by a _key_ / _value_ -pair. For convenience
531 this can be done with ArnZeroConfRegister::setTxtRecordMap() using an Arn::XStringMap.

535 Discover {#gen_discover}
536 --------
537 _ARN Discover_ is the mid level support for advertising and discovering services on
538 a local network. This implementation is only for the "arn" _service type_ and is heavily
539 dependent on the ArnLib. The "arn" _service type_ is approved and registered by IANA.
540 
541 _ARN Discover_ implementation has two parts. The ArnDiscoverAdvertise can be used to
542 advertise an %Arn _service_ given a _host address_ and a _port number_. The other part is
543 the ArnDiscoverBrowser / ArnDiscoverResolver. The browser is used to get a realtime list
544 of available %Arn _services_ on the network. The resolver is for taking a manual resolve
545 when a _service name_ is known in advance.
546 
547 _ARN Discover_ is designed to minimize external glue logic as these classes do all the
548 common processing. Internally _ARN ZeroConfig_ is used, but focus is on solving %Arn
549 specific needs in a powerful, yet flexible manner.
550 
551 An _ARN service_ needs an ArnDiscover::Type and a [service name](\ref gen_zeroconfServiceName).
552 The ArnDiscover::Type sets up a coarse division of the applications into the _groups_
553 "server" and "client". The "client" typically only offer the service of ArnDiscoverRemote.
554 
555 _ARN services_ can also have _groups_. These are identifiers that can be used to filter out
556 some sub group. An _ARN service_ can be advertised with many _groups_, but browsing can only be
557 filtered with one _group_ or with no filter.
558 
559 It's possible to add a _custom property_ to an _ARN service_. This can be done with
560 ArnDiscoverAdvertise::setCustomProperties() using an Arn::XStringMap. The propertie has a
561 _key_ / _value_ -pair. The custom property are advised to have a _key_ starting with a
562 capital letter to avoid name collision with the system.
563 The added _groups_ will be set as properties with naming as "group0", "group1" ...
564 
565 ArnDiscoverBrowser collects found %Arn _services_. Each of these _services_ can automatically
566 be further examined. This is chosen by calling ArnDiscoverBrowserB::setDefaultStopState(),
567 which e.g. tells examination to stop after _host name_ has been found. The _service_ can
568 then manually be ordered for further examination by ArnDiscoverBrowserB::goTowardState(),
569 e.g. examination should now stop after _host ip_ is found.
570 
571 All the information about a _service_ is stored in ArnDiscoverInfo. Found _services_ can
572 be accessed by index, id or _service name_. Increasing index, starting at 0, gives a list
573 of _services_ alfabetically sorted by _service name_. The index is kind of volatile and
574 should be used instantly, not be stored. The id gives a unique number for each service and
575 can be stored. However the _service_ given by the id might dissapear.

579 Discover remote {#gen_discoverRemote}
580 ---------------
581 _ARN Discover Remote_ is the highest level support for advertising and discovering services
582 on a local network. Its implementation is based on _ARN Discover_. The added functionality
583 is to have a remote control for both advertising an ArnServer and multiple ArnClient
584 connections. The remote control is done via _ARN Data Objects_ in [local path](#gen_localPath)
585 "Sys/Discover/".
586 
587 _ARN Discover Remote_ has one main class, ArnDiscoverRemote which act as a central point.
588 The ArnDiscoverRemote class also takes an ArnServer and advertises it as a _service_. For
589 remote control the _service name_ is available at [local path](#gen_localPath)
590 "Sys/Discover/This/Service/value".
591 
592 ArnDiscoverRemote can make an internal ArnServer, when there is no need to access the
593 ArnServer class. This is usually the case in an client application. The ArnServer is then
594 merely used to make the discover functionality remote controlled.
595 
596 Remote controlled client connections can be added. Each ArnClient is handled by an
597 ArnDiscoverConnector instance, which is made by ArnDiscoverRemote::newConnector().
598 Connections can be added to ArnDiscoverConnector, both as a _direct host_ list and a
599 _discover host_.
600 
601 The _discover host_ is indirerctly set, by adding an ArnDiscoverResolver to
602 ArnDiscoverConnector. A _service name_ can then be resolved into the _discover host_.
603 
604 The two connection methods can coexist and as standard the _discover host_ has lower
605 priority number than _direct host_, i.e. _discover host_ is tried first.
606 
607 The ArnDiscoverConnector is associated with an _id_, which should be chosen to describe the
608 client target or its purpose. It's not a host address or necessarily a specific host, as
609 there can be many possible addresses assigned to the ArnDiscoverConnector.
610 
611 The _id_ will appear as an _ARN folder_ in [local path](#gen_localPath), e.g. when _id_ is
612 "WeatherData-XYZ" the folder path will be "Sys/Discover/Connect/WeatherData-XYZ/". The
613 folder and its sub folders will contain _ARN Data Objects_ to remote control the ArnClient.
614 For a more comprehensive description of these objects, see
615 [help discover description](@ref helpDiscDiscover).
616 
617 In the above example, a _discover host_ can be remote controlled by setting the
618 _service name_ in [local path](#gen_localPath)
619 "Sys/Discover/Connect/WeatherData-XYZ/DiscoverHost/Service/value", e.g. to
620 "Region Weather XYZ".
621 
622 Also in the above example, the first _direct host_ can be remote controlled by setting the
623 _host name_ in [local path](#gen_localPath)
624 "Sys/Discover/Connect/WeatherData-XYZ/DirectHosts/Host-0/value", e.g. to "localhost".
625 
626 Normally it's wanted that any remote set values in the [local path](#gen_localPath) remains
627 after power cycling. This is supported by the [Arn persist system](#gen_persistArnobj).
628 
629 Connecting via resolver uses the logic:
630 * If connection fails for a _discover host_, resolving is forced to be refreshed for the
631  target _service name_. The Host for the _service name_ might have changed since last
632  resolved and doing a refresh can get the new _discover host_.
633 
634 * If connection continues to fail for a _discover host_, refreshing the resolv will have a
635  blocking time to avoid spamming the net. Typically this time is 30 seconds, but it can be
636  changed by ArnDiscoverConnector::setResolveRefreshTimeout().
