9 ScriptJobs {#int_scriptjobs}
10 ==========
11 * Each jobstack ScriptJobs is setup with a ScriptJobFactory wich makes custom interfaces etc.
12 
13 * ScriptJobControl is setup with: Sriptfile, Config (QObject) and InterfaceList.
14  Scriptfile is also copied to a ArnItem.
15 
16 * ScriptJobControl can be connected to update of script in Arn, to make reload possible.
17 
18 * Error text from ScriptJobControl can be connected to a pipe in Arn for logging.
19 
20 * ScriptJobControl together with jobpriority define the ScriptJob and is added to ScriptJobs.
21  Error text from Script job is connected to ScriptJobControl.
22 
23 * Starting ScriptJobs in cooperative mode:
24  1. Every ScriptJob is created and setup by corresponding ScriptJobControl
25  2. Every ScriptJob is connected to Scheduler (yield etc).
26  3. Every ScriptJobControl is connected to ScriptJobs for signaling update of script.
27  4. Scheduler is started.
28 
29 * Setup ScriptJob by ScriptJobControl:
30  1. set ScriptJobFactory and Config
31  2. Make and add the jobs Interfaces
32  3. Evaluate the script (in js engine)
33  4. run script function jobInit()
34 
35 * Updating Script in cooperative mode:
36  1. ScriptJobControl gets updated by Arn (or other).
37  2. ScriptJobControl sends signal to ScriptJobs, which sets an updated flag for the corresponding Script Job.
38  3. When scheduling, every updated script will get its sigQuit signal invoked and then reloaded.
39  4. Reloading includes creating a new ScriptJob and setting up with ScriptJobControl etc.
40 
41 * Starting ScriptJobs in preemtive mode:
42  1. Every ScriptJob gets its own thread which also is setup with ScriptJobControl and ScriptJobFactory.
43  2. Thread is started and it create a ScriptJobSingle where followning steps are done.
44  3. ScriptJob is created and setup by ScriptJobControl
45  4. ScriptJob is connected to Scheduler (yield etc).
46  5. ScriptJobControl is connected to ScriptJobSingle for signaling update of script.
47  6. Scheduler is started in ScriptJobSingle (just one job).
48 
49 * Updating Script in preemtive mode:
50  1. ScriptJobControl gets updated by Arn (or other).
51  2. ScriptJobControl sends signal to ScriptJobSingle, which sets an updated flag
52  and both invokes sigQuit signal to script and calls quit in scriptJob.
53  3. ScriptJob aborts its js script engine and posts a custom Quit event with high prio.
54  4. When ScriptJob get the Quit event, it will send a QuitRequest signal to ScriptJobSingle.
55  5. ScriptJobSingle will get the signal amd detect update flag, which means reloading.
56  6. Reloading includes creating a new ScriptJob and setting up with ScriptJobControl etc.

59 ArnMonitor {#int_arnmonitor}
60 ==========
61 * Monitor starts its actual connection job when its start method is called.
62 
63 * Monitor (at client-side) results in creates an ItemNet with path to monitorPath.
64 
65 * The ItemNet is also put in syncQueue (always main-thread).
66 
67 * Monitor puts the arn-event "monitorStart" in event loop,
68  which makes sure event is sent after Monitor (and its caller) has finished initiializing.
69 
70 * When "monitorStart" is received on local (client) side, the ItemNet will change SyncMode to Monitor.
71  This will resync ItemNet to a Monitor at any server restart.
72 
73 * Now 2 possibilities depending on threading:
74  1. The ItemNet was sent before syncMode Monitor was set. Then server will receive an ordinary
75  ItemNet and do standard setup.
76  2. The ItemNet was sent with syncMode Monitor set. The server will detect this and do
77  MonitorSetup on the ItemNet.
78 
79 * When arn-event "monitorStart" is received on server-side, if SyncMode is not already set to "Monitor",
80  server will do MonitorSetup on the ItemNet.
81 
82 * When doing MonitorSetup (at server-side), logic are made to send arn-events when new
83  childs are created, and present childs are directly sent as arn-event.

86 Destroy {#int_destroy}
87 =======
88 * Destruction can be locally initiated and affects one link. Destruction can be set as
89  local or global.
90 
91 * Destruction can also be initiated for leaves by the destroy command and arives with a netId.
92  Or it can be with the delete command for a folder (tree).
93 
94 * For leaf, corresponding ItemNet is disabled (set as defunct), which prohibit sending destroy
95  command back to the originator of the command.
96 
97 * The ItemNet is also destroyed in the same way as a locally initiated destruction and affects
98  one link. Destruction is set to be global.
99 
100 * The affected link:s tree is recursively traversed and all links are first marked as retired.
101  Also the retire type is set as LeafLocal, LeafGlobal or Tree.
102 
103 * As the last thing in this recursion each link is sending a Retired ArnEvent, ie the leaves are
104  the first to send. The event is sent to the subscriptions (ArnBasicItems or derived) of each
105  link.
106 
107 * If it's a destroy of a tree (folder), a Retired ArnEvent is also sent to the tree:s parent
108  and all the way up to the root. The event is sent to the subscriptions (ArnBasicItems) of each
109  link. These events have a marking telling destroy is below.
110 
111 * The Retired ArnEvent is handled by each subscribing Item. For ArnBasicItem this is done by
112  its eventhandler, which by default is an internal handler. For ArnItem this is done by
113  sending a linkDestroyed signal to be handled by application code.
114  The Items is finally closed and by this the link ref counter is decremented.
115 
116 * When the links ref counter is reaching zero, a ZeroRef ArnEvent is sent. Also a ZeroRef pending
117  counter is increased.
118 
119 * The event is handled by ArnM::doZerRefLink(), in Main thread. First the ZeroRef pending
120  counter is decreased. Next both ref counter and ZeroRef pending counter is checked to be zero,
121  which indicates that this is the final ZeroRef for this link.
122  This is to prohibit a scenario where the link has been reused during ZeroRef ArnEvent delivery.
123  Also this reuse might have been followed by a dropped usage resulting in a second ZeroRef
124  ArnEvent.
125 
126 * In ArnM::doZerRefLink() if this is the final ZeroRef, it will set the link ref counter
127  to -1, to mark the link as fully de-referenced.
128  The link and parent (and grand parants ...) are deleted if they don't have any children
129  and ref = -1 and they are marked retired.
130 
131 * When the ArnSync, which is eventHandler for ItemNet, is handling the Retired ArnEvent, it
132  will delete the corresponding ItemNet from sync map and all queues. Finally a command can
133  be sent with its netId.
134 
135 * The sent command depends on retire type. For Leaflocal, a nosync command is used.
136  For LeafGlobal, a delete command is used to spread the destruction to server and other
137  clients. The Tree type doesn't send a command at item level.
138 
139 * For tree destroy, ArnClient is using a monitoring ArnItemNetEar at each mount point to catch
140  the Retire ArnEvent for a tree below. Such an event is resulting in a delete or noSync
141  command is sent, depending on global or local destroy. The command is sent with the path
142  to the destroyed tree.
143 
144 * For tree destroy, ArnServer is using a monitoring ArnItemNetEar at root to catch the Retire
145  ArnEvent for a tree below. Such an event is resulting in a delete command is sent.
146  The command is sent with the path to the destroyed tree.
147 
148 * When a delete command is echoed back to the originator, it will stop with this only echo
149  as the affected tree is already marked for retire and this will terminate the command.
